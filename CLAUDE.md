# Инструкции для работы в этом проекте

Эти правила снижают типичные ошибки LLM при написании кода. Объединять с
project-specific инструкциями по необходимости.

Компромисс: правила смещают баланс в сторону осторожности, а не скорости.
Для тривиальных задач — действовать по здравому смыслу.

## 1. Думай перед кодом
Не предполагай молча. Не прячь сомнения. Проговаривай компромиссы.

Перед реализацией:
- Явно озвучивай свои допущения. Если не уверен — спрашивай.
- Если возможно несколько трактовок — показывай их, не выбирай молча.
- Если есть более простой подход — скажи. Возражай, когда это оправдано.
- Если что-то неясно — остановись. Назови, что именно непонятно. Спроси.

## 2. Сначала простота
Минимум кода, который решает задачу. Ничего спекулятивного.

- Никаких фич сверх запрошенного.
- Никаких абстракций для одноразового кода.
- Никакой «гибкости» или «конфигурируемости», которую не просили.
- Никакой обработки невозможных сценариев.
- Если написал 200 строк, а можно 50 — перепиши.

Спроси себя: «Senior-инженер назвал бы это переусложнённым?» Если да — упрости.

## 3. Хирургические изменения
Трогай только то, что должен. Убирай только свой мусор.

При редактировании существующего кода:
- Не «улучшай» соседний код, комментарии, форматирование.
- Не рефактори то, что не сломано.
- Соблюдай существующий стиль, даже если сделал бы иначе.
- Заметил несвязанный мёртвый код — упомяни, не удаляй.

Когда твои изменения создают «сирот»:
- Удали импорты/переменные/функции, которые стали неиспользуемыми ИЗ-ЗА твоих правок.
- Не удаляй ранее существовавший мёртвый код без запроса.

Тест: каждая изменённая строка должна напрямую следовать из запроса пользователя.

## 4. Исполнение от цели
Определи критерии успеха. Повторяй, пока не проверено.

Превращай задачи в проверяемые цели:
- «Добавь валидацию» → «Напиши тесты на невалидный ввод, потом сделай, чтобы прошли»
- «Исправь баг» → «Напиши тест, воспроизводящий баг, потом сделай, чтобы прошёл»
- «Отрефактори X» → «Убедись, что тесты проходят до и после»

Для многошаговых задач — кратко озвучь план:
1. [Шаг] → проверка: [как проверю]
2. [Шаг] → проверка: [как проверю]

Сильные критерии успеха позволяют работать автономно. Слабые («просто чтобы
работало») требуют постоянных уточнений.

Правила работают, если: меньше лишних изменений в диффах, меньше переписываний
из-за переусложнения, а уточняющие вопросы идут ДО реализации, а не после ошибок.
